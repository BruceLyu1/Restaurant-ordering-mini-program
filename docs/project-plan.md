# 香港餐厅扫码点餐系统项目计划书

更新日期：2026-06-03

## 1. 项目概述

本项目是一个面向香港本地中小餐厅的扫码点餐 H5 系统。目标是解决餐厅仍然依赖人工纸笔点餐的问题，让顾客通过桌面二维码进入手机点餐页面，自助浏览菜单、加入购物车、确认下单；餐厅后台则可以按桌号接收订单、打印厨房小票、结账，并统计每日热门菜品和销售数据。

当前版本是前端 H5 原型，已经可以完成从顾客扫码点餐到餐厅后台处理订单的核心流程。数据目前保存在浏览器本地，用于快速演示和验证业务流程；正式商用版本需要接入数据库、员工登录、图片存储、实时同步和打印服务。

## 2. 项目目标

### 2.1 用户目标

- 顾客扫码即可进入对应桌号的点餐页面。
- 顾客可以按分类查看菜品，选择菜品数量，加入购物车。
- 顾客可以在下单前修改或取消误点菜品。
- 顾客确认下单后，可以随时查看本桌订单明细和订单状态。
- 后台结账后，本桌订单在顾客端清空，方便下一批顾客使用。

### 2.2 餐厅目标

- 餐厅后台可以看到每张桌提交的订单。
- 订单按顾客下单顺序排列，厨房按顺序出单。
- 后台可以手动打印或自动打印厨房小票。
- 前台可以结账，结账后订单进入已完成状态。
- 管理员可以维护菜品、分类、图片、价格、售罄状态和供应时段。
- 管理员可以查看销量排行、营业额统计和热门菜品。

### 2.3 商业目标

- 以低成本 H5 方式服务香港餐厅，不强制餐厅安装复杂系统。
- 让餐厅可以快速试用，先验证扫码点餐是否提升运营效率。
- 后续可扩展为多餐厅 SaaS 系统，按餐厅或按门店收费。

## 3. 当前已完成功能

### 3.1 顾客点餐端

- 手机 H5 点餐界面。
- 顾客端和后台端入口切换。
- 根据 URL 参数识别桌号，例如 `?view=guest&table=03`。
- 每张桌显示独立桌号。
- 菜品分类展示，包括饭类、点心、面类、小菜、甜品、饮品等。
- 推荐分类按顾客累计点餐量排序。
- 菜品展示真实菜品图片、名称、描述和价格。
- 加入购物车、增加数量、减少数量、删除菜品。
- 下单前购物车确认。
- 下单后显示订单号、桌号和金额。
- 顾客端有“订单详情”入口，可以查看本桌已提交订单。
- 订单详情显示菜品明细、数量、金额和状态。
- 后台结账后，本桌订单不再显示，方便翻台。
- 售罄菜品和非当前供应时段菜品不可下单。

### 3.2 餐厅后台

- 营运总览。
- 订单管理。
- 菜单管理。
- 桌位管理。
- 报表分析。
- 员工管理。
- 打印设置。
- 餐厅设置。

### 3.3 订单管理

- 按下单顺序显示订单。
- 区分待处理、已列印、已结账状态。
- 支持手动列印。
- 支持结账。
- 自动打印开关打开后，新订单会直接进入已列印状态。
- 后台订单右侧显示今日热门菜式。
- 热门菜式按当天订单销量实时排序。

### 3.4 菜单管理

- 新增菜品。
- 修改菜品名称、价格、分类、图片和供应时段。
- 删除或下架菜品。
- 上传或更换菜品图片。
- 切换售罄状态。
- 按菜品名称搜索。
- 按分类筛选菜品。
- 新增菜品时可以选择已有分类或新增分类。
- 支持分类修改、合并和删除，避免重复分类。
- 支持早餐、午市、晚市不同供应时段。

### 3.5 桌位管理与二维码

- 查看所有桌位。
- 切换桌位空闲或使用中状态。
- 每张桌可以生成对应点餐链接。
- 每张桌可以查看二维码桌码。
- 顾客扫码后进入对应桌号的点餐页面。
- 二维码弹窗支持复制链接和列印桌码。

### 3.6 报表分析

- 今日营业额。
- 本周营业额。
- 本月营业额。
- 本年营业额。
- 订单总数。
- 售出餐点总数。
- 菜品销量排行。
- 菜品销售额统计。
- 热门菜式可以跳转到完整报表。

### 3.7 打印设置

- 选择打印机。
- 设置每张订单份数。
- 自动打印新订单开关。
- 新订单提示声开关。
- 打印测试小票。

### 3.8 餐厅设置

- 修改餐厅名称。
- 修改电话。
- 修改地址。
- 设置默认语言。
- 设置早餐、午市、晚市供应时间。

## 4. 当前技术架构

### 4.1 前端

- React
- Vite
- 原生 CSS
- H5 响应式页面
- GitHub Pages 静态部署

### 4.2 数据存储

当前使用浏览器 `localStorage` 模拟数据存储，包括：

- 菜单数据
- 订单数据
- 桌位数据
- 员工数据
- 餐厅设置
- 打印设置

这种方式适合演示和原型验证，但不适合正式商用，因为数据只保存在当前浏览器，换设备后不会同步。

### 4.3 部署

- 代码仓库：GitHub

- 当前线上地址：
  - 顾客端：`https://brucelyu1.github.io/harbour-ordering-h5/?view=guest`
  - 后台端：`https://brucelyu1.github.io/harbour-ordering-h5/?view=admin`

## 5. 当前限制

当前版本还不是正式商用系统，主要限制包括：

- 数据只保存在浏览器本地，无法多设备同步。
- 没有真实后台数据库。
- 没有员工登录和权限控制。
- 没有多餐厅、多门店隔离。
- 菜品图片上传后只保存在当前浏览器。
- 没有真实打印机接口，只是模拟打印流程。
- 二维码目前用于生成桌号点餐链接，正式版还需要绑定餐厅和门店。
- 暂时不做在线支付、会员、优惠券和外卖功能。

## 6. 下一阶段开发路线图

### 阶段一：试用版基础后台

目标：让一家餐厅可以真实试用。

- 接入 Supabase 数据库。
- 建立餐厅、桌位、菜品、订单、员工数据表。
- 员工登录后台。
- 菜单数据从数据库读取。
- 顾客下单写入数据库。
- 后台实时收到订单。
- 菜品图片上传到云端存储。
- 二维码链接绑定餐厅 ID 和桌号。

### 阶段二：厨房出单与前台结账

目标：让餐厅可以在真实营业环境测试。

- 接入真实打印服务。
- 支持厨房小票格式配置。
- 支持自动打印失败后的补打。
- 后台订单状态实时同步到顾客端。
- 前台结账后清空该桌当前订单。
- 增加订单历史查询。
- 增加日结统计。

### 阶段三：多餐厅 SaaS 版本

目标：让系统可以服务多家餐厅。

- 支持多餐厅注册。
- 支持多门店。
- 每家餐厅独立菜单、订单、员工和报表。
- 不同员工角色权限控制。
- 系统管理员后台。
- 餐厅套餐和订阅管理。
- 数据备份和异常日志。

### 阶段四：商业增强功能

目标：提高产品竞争力，但不作为当前优先级。

- 在线支付。
- 会员系统。
- 优惠券。
- 外卖自取。
- 多语言菜单。
- 菜品库存。
- 套餐和加料选项。
- 后厨屏幕 KDS。

## 7. Supabase 接入计划

建议使用 Supabase 作为第一版后端，原因是它能同时提供数据库、登录鉴权、图片存储和实时数据同步，适合当前 React H5 快速落地。

### 7.1 需要建立的数据表

- `restaurants`：餐厅资料。
- `tables`：桌位资料和二维码绑定。
- `menu_categories`：菜品分类。
- `menu_items`：菜品资料。
- `orders`：订单主表。
- `order_items`：订单菜品明细。
- `staff`：员工资料。
- `printer_settings`：打印设置。
- `restaurant_settings`：餐厅设置和营业时段。

### 7.2 登录权限

建议先做三种角色：

- 老板/经理：可以管理菜单、员工、报表和设置。
- 前台：可以查看订单、列印、结账。
- 厨房：可以查看和处理订单，但不能修改菜单和报表。

### 7.3 图片存储

菜品图片上传到 Supabase Storage。后台上传图片后，顾客端和后台端都从同一个图片地址读取，避免当前本地图片无法多设备同步的问题。

### 7.4 实时同步

使用 Supabase Realtime：

- 顾客下单后，后台订单列表实时出现。
- 后台列印后，顾客端订单状态变为已列印。
- 后台结账后，顾客端本桌订单清空。

## 8. 餐厅试用方案

### 8.1 试用对象

建议先找 1 到 3 家香港本地小型餐厅试用，例如：

- 茶餐厅
- 港式小馆
- 粥粉面店
- 点心店

### 8.2 试用准备

- 为餐厅录入真实菜单。
- 上传真实菜品照片。
- 设置桌位数量。
- 为每张桌打印二维码桌码。
- 培训前台如何查看订单、列印和结账。
- 培训厨房如何按订单准备餐点。

### 8.3 试用流程

1. 顾客入座。
2. 顾客扫描桌面二维码。
3. 顾客在手机上点餐。
4. 后台收到订单。
5. 厨房按顺序准备。
6. 顾客可继续加菜。
7. 用餐结束后前台结账。
8. 后台查看当天销售数据。

### 8.4 试用观察指标

- 顾客是否愿意扫码点餐。
- 顾客是否能顺利完成下单。
- 员工是否觉得后台操作简单。
- 厨房是否能按订单顺序出餐。
- 是否减少人工抄单错误。
- 是否提高翻台速度。
- 菜品销量报表是否对餐厅有帮助。

## 9. 商业落地步骤

### 9.1 MVP 验证

先用当前 H5 原型和 1 家餐厅沟通，验证餐厅是否真的需要这个系统，以及愿意为哪些功能付费。

### 9.2 试点版本

接入 Supabase 后，选择一家餐厅做真实试点。试点期间重点观察订单同步、打印、结账和顾客体验。

### 9.3 收费方式建议

可考虑以下模式：

- 月费制：每家餐厅每月固定费用。
- 门店制：每个门店一个价格。
- 基础版免费，高级功能收费。
- 初期可以提供低价试用，换取真实反馈和案例。

### 9.4 推广方式

- 直接拜访香港本地小餐厅。
- 准备一页式介绍材料。
- 用线上演示链接给餐厅老板看。
- 打印几张真实桌码现场演示。
- 强调不需要顾客下载 App。
- 强调可以减少人工点餐和抄错单。

## 10. 优先级建议

### 当前最应该做

1. 接入 Supabase 数据库。
2. 员工登录后台。
3. 菜品图片云端上传。
4. 订单实时同步。
5. 正式二维码桌号绑定。
6. 打印服务对接。

### 当前不建议优先做

- 在线支付。
- 会员系统。
- 优惠券。
- 外卖功能。

这些功能会增加复杂度，但对第一阶段验证扫码点餐价值不是必需。

## 11. 项目定位

本项目当前定位为：

> 面向香港中小餐厅的低成本扫码点餐 H5 系统原型。

下一阶段目标是：

> 从前端演示原型升级为可供真实餐厅试用的云端扫码点餐系统。

最终目标是：

> 打造成一个可服务多家餐厅的扫码点餐 SaaS 平台。

## 12. 当前结论

目前项目已经完成了最核心的产品原型：顾客扫码点餐、后台收单、菜单管理、桌位二维码、订单状态、报表统计和基础设置。它已经具备拿给餐厅老板演示的基础。

接下来真正关键的不是继续堆更多界面功能，而是接入真实后端和真实数据同步，让系统从“可演示”进入“可试用”。只要完成 Supabase、员工登录、图片存储和订单实时同步，就可以开始找第一家餐厅做小范围试点。
